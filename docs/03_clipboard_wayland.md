# 03. Работа с буфером Wayland и эмуляция ввода

Этот документ описывает, как взаимодействовать со средой Fedora 44 KDE Plasma (Wayland) в условиях жестких ограничений безопасности.

## 1. Мониторинг буфера обмена (Слой Демона)
- Использовать крейт `smithay-clipboard` для подключения к Wayland-протоколу `wlr-data-control`.
- Демон должен работать в асинхронном потоке (`tokio::task`), отслеживая появление новых данных в `SELECTION` или `PRIMARY` буферах.
- **Дедупликация:** Перед записью в БД, текст/картинка проверяются. Если хэш (MD5/SHA256) совпадает с самым последним элементом в базе данных — запись игнорируется.

## 2. Механизм вставки элемента в активное окно
Когда пользователь кликает на элемент в UI или нажимает `Alt+1`..`Alt+9`, происходит следующий цепочечный процесс:

1. UI-клиент сообщает Демону ID выбранного элемента по IPC.
2. Демон извлекает данные из БД и обновляет системный буфер Wayland (делает этот текст текущим активным буфером обмена).
3. UI-клиент (`maccy-kde --popup`) мгновенно завершает свою работу и закрывает окно, возвращая фокус на предыдущее активное приложение (текстовый редактор, браузер и т.д.).
4. **Эмуляция нажатия:** Демон выдерживает паузу в 50-100 мс (чтобы фокус гарантированно вернулся на окно) и программно имитирует нажатие комбинации клавиш **`Ctrl+V`**.

### Реализация эмуляции ввода в KDE Plasma 6 (Wayland):
В Wayland стандартные инструменты (вроде старого `xdotool`) заблокированы. ИИ-агент должен реализовать эмуляцию через **DBus-интерфейс KWin**.
- Использовать крейт `zbus` для вызова метода виртуальной клавиатуры KDE: `org.kde.kwin.VirtualKeyboard`.
- Альтернативный рабочий вариант для Flatpak среды — использование портала `org.freedesktop.portal.RemoteDesktop`, запрашивая виртуальный ввод клавиатуры.
